# How To Document

This document defines the minimum structure for project documentation.

## Documentation Rules

- write user outcome first, technical detail second
- keep scope and ownership explicit
- separate rules from ideas and from history
- avoid vague or aspirational language without contract

## Minimum Structure

Each non-trivial document should include:

1. purpose
2. scope
3. contract or rules
4. operational implications
5. change policy

## Change Discipline

- when behavior changes, update the owning document in the same work item
- when architecture boundaries change, update `governance/app-architecture/ARCHITECTURE.md`
- when reusable baseline changes, update `governance/app-architecture/architecture-standard.md`
